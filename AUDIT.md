## Project Goals

The project is an Arkham Horror-themed multiplayer web game targeting intermediate developers
learning WebSocket client-server architecture. It promises:

1. **5 core game mechanics**: Location System, Resource Tracking, Action System, Doom Counter, Dice Resolution.
2. **Multiplayer**: 1-6 concurrent players; late join; real-time state sync; auto-reconnection.
3. **Go server**: Interface-based design (`net.Conn`, `net.Listener`, `net.Addr`); goroutine/channel concurrency; explicit error handling.
4. **Multi-platform clients**: Legacy HTML/JS canvas client; Ebitengine desktop, WASM, and mobile (alpha).
5. **Observability**: Prometheus-compatible `/metrics`, health check at `/health`, real-time dashboard at `/dashboard`.
6. **Performance**: Sub-500 ms state sync, sub-100 ms health check, stable at 6 players for 15+ minutes.
7. **JSON protocol**: 5 message types — `gameState`, `playerAction`, `gameUpdate`, `connectionStatus`, `diceResult`.
8. **Win/Lose**: Win on 4 clues × player-count via Act deck; lose at doom 12.
9. **Session persistence**: Server issues reconnect tokens; JS client reclaims slots. (Ebitengine client noted as not yet supporting token-based reclaim in README — however README misdescribes the current JS capability.)

## Findings

### CRITICAL

- [ ] **Ebitengine client always receives empty game state (protocol mismatch)** — `client/ebiten/net.go:3-18`, `cmd/server/game_server.go:1636-1643` — The server broadcasts `{"type":"gameState","data":{...full GameState...}}`. The Ebiten `serverMessage` struct reads `Players`, `CurrentPlayer`, `Doom`, etc. as **top-level JSON fields**. Because these fields live under the `"data"` key (not at the root), every field decodes to its zero value: `Players` = nil, `Doom` = 0, `GamePhase` = `""`. The desktop and WASM clients build and connect successfully but render a permanently empty/blank board. The comment on `serverMessage` ("The server's gameState broadcast inlines all fields at the top level") is factually incorrect. **Remediation:** Either (a) change `broadcastGameState()` to emit fields at the root (matching the `serverMessage` struct) — i.e., marshal `gs.gameState` directly into the top-level JSON object; or (b) change `decodeGameState()` to unmarshal `msg.Data` into a `GameState` struct instead of reading top-level fields. Option (b) is safer as it matches the existing JS client behaviour (`this.gameState = message.data`). **Validation:** `go test -race ./cmd/server/...` plus a new unit test `TestDecodeGameState_FromDataWrapper` in `client/ebiten/net_test.go`.

- [ ] **`ConnectionWrapper.Read()` reports wrong byte count on truncation** — `cmd/server/connection_wrapper.go:37-42` — `copy(b, data)` silently truncates if `len(b) < len(data)`, but the function returns `len(data)` unconditionally. Any caller that checks the `n` return value receives an inflated count, leading to incorrect slice operations or silent data loss. The `net.Conn` contract requires `n == len(b)` only when no error occurs and enough buffer space exists; returning `len(data)` when fewer bytes were actually written violates this contract. **Remediation:** Replace `return len(data), nil` with `n = copy(b, data); return n, nil`. If `len(b) < len(data)`, consider also returning `io.ErrShortBuffer`. **Validation:** `go test -race ./cmd/server/...`; add `TestConnectionWrapper_ReadTruncation` checking `n == min(len(b), len(data))`.

### HIGH

- [ ] **`drawMythosToken()` is deterministic, not random** — `cmd/server/game_server.go:672-674` — The Mythos cup token is drawn using `tokens[gs.gameState.Doom % len(tokens)]`. At doom=0 the token is always `"doom"`, doom=1 always `"blessing"`, doom=2 always `"curse"`, doom=3 always `"blank"`, doom=4 always `"doom"`, etc. Players can trivially predict every Mythos Phase outcome. The AH3e theme and the randomness claims in the codebase are both violated. The function is named "pseudo-random" in its comment but the implementation has no randomness at all. **Remediation:** Use `mathrand.Intn(len(tokens))` — the same `mathrand` import already used in `rollDice()` — to select the token index: `return tokens[mathrand.Intn(len(tokens))]`. **Validation:** Statistical test asserting all four tokens appear across ≥100 draws; run with `go test -race ./cmd/server/...`.

- [ ] **Ebitengine client cannot reclaim player slot on reconnect** — `client/ebiten/net.go:57-73` (reconnectLoop) and `client/ebiten/net.go:209-218` (applyConnectionStatus) — The `ConnectionStatusData` struct decoded from `connectionStatus` messages has no `Token` field, so the server-issued reconnect token is discarded. `reconnectLoop` dials with no `?token=` query parameter. As a result, every reconnect creates a new player slot (new ID, empty resources, appended to turn order) while the old disconnected slot persists as a zombie. The JS client correctly stores and resends the token (`game.js:9,151,71`). **Remediation:** Add `Token string \`json:"token"\`` to `ConnectionStatusData`; store it in `LocalState`; append `?token=<token>` to the dial URL in `reconnectLoop()` when a token is available. **Validation:** Integration test verifying reconnected Ebitengine client reuses the same player ID.

- [ ] **README §Connection Behaviour incorrectly states session persistence is not implemented** — `README.md:110` — The README says "a disconnected player cannot reclaim their investigator. Reconnecting after a drop creates a new player slot." This is false for the JS client, which has full token-based slot restoration (`game.js:70-71,151`), and for the server, which has `restorePlayerByToken()` (`game_server.go:976`). The statement is only correct for the Ebitengine client (see HIGH-3 above). **Remediation:** Replace README:110 with: "The JS client reclaims its player slot automatically using a server-issued reconnect token. The Ebitengine client does not yet implement token reclaim; reconnecting creates a new slot (tracked in GAPS.md)." **Validation:** Manual smoke test: connect legacy JS client, close tab, reopen → same player ID returned.

- [ ] **`performComponent()` accepts action type but always returns an error** — `cmd/server/game_server.go:540-547` — `ActionComponent` (`"component"`) is listed in `isValidActionType()` as a valid action and is dispatched by `dispatchAction()`, but `performComponent()` always returns `fmt.Errorf("component action … not yet implemented")`. From the client's perspective, sending a `"component"` action receives an error response (no state update, no dice result, no feedback beyond server log), which is confusing. Alternatively, the action should be removed from `isValidActionType()` until implemented. **Remediation:** Either (a) remove `ActionComponent` from the slice in `isValidActionType()` so the server rejects it early with "invalid action type"; or (b) implement the action fully. Option (a) is the safer short-term fix. **Validation:** `TestProcessAction_InvalidActionType` should cover the rejection path after fix.

### MEDIUM

- [ ] **`validateResources()` cyclomatic complexity 17.4 with copy-paste duplication** — `cmd/server/game_server.go:129-165` — The function is flagged at complexity 17.4 (highest in the codebase) and `go-stats-generator` identifies a 36-line clone at lines 130-147 and 148-165 (duplicate clamp patterns for each resource field). The structure `if x < 0 { x = 0 } if x > max { x = max }` repeats 6 times. This inflates complexity metrics and creates a maintenance risk when adding new resource types. **Remediation:** Extract a `clampInt(v, lo, hi int) int` helper and refactor `validateResources` into a table-driven loop over `(field, min, max)` triplets. **Validation:** `go test -race ./cmd/server/...` — all `TestValidateResources_ClampsBounds` subtests must pass.

- [ ] **`game_server.go` is 1591 lines with 77 functions** — `cmd/server/game_server.go` — Flagged as oversized by `go-stats-generator` (burden score 2.71, highest in the codebase). Game mechanics, network handling, observability, error recovery, and state management are co-located. `go-stats-generator` identifies 70 misplaced functions. **Remediation:** Extract into packages: `game_mechanics.go` (rollDice, perform*, advanceTurn, checkGameEnd), `observability.go` (metrics, health, dashboard handlers), `connection.go` (handleWebSocket, registerPlayer, runMessageLoop). This is a refactoring-only change. **Validation:** `go build ./cmd/server && go test -race ./cmd/server/...`.

- [ ] **README §Connection Behaviour describes a reconnection *window* but code implements an *inactivity* timeout** — `README.md:107-108`, `cmd/server/game_server.go:896-914` — The README says "The server applies a **30-second inactivity timeout**: if no message arrives from a connected player within 30 seconds, the doom counter is incremented and the connection is closed." This is accurately described. However, the README's earlier sentence ("**30-second reconnection timeout**") appears in the Troubleshooting §Connection Issues section in a way that implies a grace window. The `runMessageLoop()` sets a `SetReadDeadline` of 30 seconds on *every* iteration of the read loop, meaning the deadline is reset after every received message. The initial `handleConnection()` also sets a 30-second deadline before the WS handshake, which *is* a connection timeout (not an inactivity timeout). These two distinct timeouts share the same value and are conflated in the documentation. **Remediation:** Add a code comment to `handleConnection()` distinguishing "handshake timeout (30 s)" from the "per-message inactivity timeout (30 s renewed on receipt)". Update README to clarify that the 30-second timeout is an *inactivity* deadline, not a reconnection window. **Validation:** No code changes required; doc-only fix.

- [ ] **`totalGamesPlayed` counter can double-increment** — `cmd/server/game_server.go:747-790`, `checkActAdvance()`, `checkAgendaAdvance()` — `checkGameEndConditions()` has no guard for `gs.gameState.GamePhase == "ended"` at its start. If called after the game has already ended (e.g., from a Mythos Phase doom increment that follows a tentacle-roll doom increment that also hit the cap), the counter is atomically incremented again. The doom-cap `>= 12` check returns early with its own `atomic.AddInt64`, but if another code path calls `checkGameEndConditions` while `GamePhase == "ended"` the agenda/act paths will also trigger. **Remediation:** Add an early return at the top of `checkGameEndConditions()`: `if gs.gameState.GamePhase == "ended" { return }`. **Validation:** Unit test that calls `checkGameEndConditions()` twice on an already-ended game and asserts `totalGamesPlayed == 1`.

### LOW

- [ ] **`go-stats-generator` reports 12 unreferenced functions** — various files — Dead code inflates the binary and creates maintenance confusion. Most appear to be helper constructors or metric aggregators that are only called from test files (not counted by `--skip-tests` mode). **Remediation:** Run `go-stats-generator analyze . --format json | jq '.maintenance.dead_code'` after addressing the above findings to identify which remain; remove confirmed dead code or add `_test.go` suffix to test-only helpers. **Validation:** Re-run `go-stats-generator analyze . --skip-tests`.

- [ ] **`min`/`max` shadow Go 1.21 builtins** — `cmd/server/game_utils.go:9-20` — Go 1.21 introduced built-in `min` and `max` functions. The project targets Go 1.24.1 (`go.mod:3`), so these local implementations shadow the builtins. `go vet` does not currently warn about this, but future compiler updates may. **Remediation:** Remove `game_utils.go:9-20` and rely on the built-in `min`/`max`. Verify all call sites compile. **Validation:** `go build ./cmd/server/...`.

- [ ] **Ebitengine `client/ebiten` package has zero test coverage** — `client/ebiten/` — `go-stats-generator` found no test files in the `ebiten` or `render` packages. Given that CRITICAL-1 (protocol mismatch) was undetected, the absence of round-trip tests is a systemic risk. **Remediation:** Add `client/ebiten/net_test.go` with at minimum `TestDecodeGameState_FromDataWrapper` and `TestApplyConnectionStatus_PreservesToken`. **Validation:** `go test ./client/ebiten/...`.
